Sirno 检查结构。

结构检查包括必需的元数据字段、接受的字段形状、
引用存在性、生成页脚边界，
以及在请求时的见证查找有效性。
它们通过 `[structural]` 下配置的字段检查引用。
当配置了 `[repo].members` 时，
评审检查报告命名了缺失条目的仓库见证块。

生成链接检查有两个部分。
哨兵形状始终被检查。
新鲜度由 `[check].link` 控制，
该选项默认启用。

在编辑期间，悬挂的结构 ID 可能警告。
在显式评审边界，悬挂引用是错误。
不在 `[structural]` 中的列表值元数据字段始终警告。

检查保持局部移动快速，同时使评审边界严格。
它们不决定散文是否为真，或代码是否满足声明。

检查边界重要，因为编辑和评审有不同的需求。
在编辑期间，人可以创建一个条目而不创建所有相关目标。
警告保持工作可见而不阻塞移动。
在评审时，悬挂引用应该被修复，
因为 Lake 被视为一个连贯的设计形式。

文件检查保持公开 Lake 形状可预测。
条目目录包含具有有效 ID 的 Markdown 条目文件。
每个文件以接受的前言开始。
生成页脚哨兵必须格式正确。
格式错误的结构值是错误，因为工具无法从中安全推断目标 ID。
未配置的结构字段是警告，因为条目命名的结构项目配置未启用。

元数据目标检查保持图可导航。
如果一个条目通过已配置的结构元数据命名一个目标，
该目标应该在 Lake 被评审时存在。
这让查询结果、生成页脚和读者导航就同一组条目达成一致。

语义评审仍然是人和智能体的工作。
检查器可以说一个见证块形状正确。
它可以报告引用缺失条目 ID 的已配置见证块。
它不能说被见证的仓库工件是声明的良好实现。
这种区分保持 Sirno 有用而不假装解决判断问题。
