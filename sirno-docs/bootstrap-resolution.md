This narrative serves readers who feel the recursive pressure in this *lake*:
Sirno is the tool being explained,
and Sirno is also the method used to explain it.

The conversation began with a simple discomfort.
A sentence could describe how any Sirno-managed project should document itself.
The next sentence could describe how this repository uses Sirno to document Sirno.
Both were valid,
but the role shift made the reader do hidden work.

The first answer was perspective.
The *lake* now names four speaking positions:
`Sirno`, `a Sirno-managed project`, `this repository`, and `this lake`.
A reader can tell whether a passage is about the general tool,
any project that applies it,
the implementation workspace,
or the self-hosted Sirno Lake in `sirno-docs/`.

The second answer was vocabulary.
Sirno terms become proper names when they appear with Sirno:
Sirno Lake and Sirno Frost.
Otherwise they stay lowercase and italic when they carry Sirno meaning:
*lake*, *entry*, *witness*, *frost*, *ripple*, *transform*, *monograph*, and *repository*.
Ordinary project words remain plain.
This keeps normal documentation readable
while still marking when the reader has entered Sirno's model.

The decisive answer was `meta`.
The bootstrap problem needed a place where the project could answer:
how should documentation develop here?
The `meta` category gives that question an address.
A `meta` *entry* can define vocabulary,
term styling,
narrative habits,
*entry* splitting rules,
and agent-facing documentation method.

That made the recursion productive.
Sirno does not just give the project documentation.
It gives the project a place to document how documentation should grow.
The method can be read,
queried,
reviewed,
and revised by the same tools that use it.

The route through the resolution is:

The implementation keeps the same conclusion in the *lake*,

Carry this aftertaste:
